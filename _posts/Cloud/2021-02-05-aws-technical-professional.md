## M2. AWS 핵심 기술

### AWS 클라우드 소개

**클라우드란?**<br />
기존의 온프레미스 IT 환경에서는 비즈니스 시스템을 실행하는데 필요한 모든 물리적 구성 요소를 회사에서 소유, 운영, 유지 관리 및 보관했음.<br />
사용자는 기업 네트워크에 연결하고 로그온한 다음 리소스에 액세스 함.<br />
클라우드는 클라우드 서비스 픓랫폼에서 호스팅하는 인터넷을 통해 동일하거나 유사한 리소스에 대한 액세스를 제공함.

**클라우드 컴퓨팅이란**<br />
클라우드 컴퓨팅은 컴퓨팅 파워, 데이터베이스, 스토리지, 애플리케이션 및 기타 IT 리소스를 인터넷을 통해 온디맨드로 제공하고 사용한만큼 비용을 지불하는 것.

**클라우드 컴퓨팅의 이점**<br />
1. 자본 비용을 가변 비용으로 대체
1. 규모의 경제로 얻게 되는 이점
1. 용량 추정 불필요
1. 속도 및 대응력 개선
1. 데이터 센터 운영 및 유지 관리에 비용 투자 불필요
1. 몇 분 만에 전 세계에 배포

**AWS 클라우드 소개**<br />
온디맨드 방식으로 초단위로 제공되며, 요금은 종량제로 부과되는 다양한 글로벌 클라우드 기반 서비스의 폭넓은 포트폴리오를 가짐<br />
*컴퓨팅, 스토리지, 데이터베이스, 분석, 네트워킹, 모바일, 개발자 도구, 관리 도구, IoT, 보안, 엔터프라이즈 앱*

### 글로벌 인프라

**AWS 글로벌 인프라: 리전**<br />
Amazon의 클라우드 컴퓨팅 리소스가 전 세계 여러 위치에서 호스팅 되는 위치를 AWS 리전이라고 함<br />
각 리전은 다른 리전에서 완전히 결리되도록 설계 -> 가장 강력한 내결함성 및 안정성 달성

**AWS 글로벌 인프라: 가용 영역**<br />
가용 영역(AZ)라는 서로 격리되어 있는 여러 개의 지리적 영역이 각 리전 내에 구성됨<br />
각 AZ는 리전 내의 다른 AZ와는 완전히 격리됨<br />
AZ는 지리적 특징을 고려해 물리적으로 분리되어 있지만, 한 리전 내의 모든 AZ는 고대역폭, 짧은 지연 시간의 완전 이중화 광섬유를 통해 상호 연결됨

**AWS 글로벌 인프라: PoP (Point of Presence)**<br />
PoP는 엣지 로케이션과 리전별 엣지 캐시 서버로 구성<br />
이러한 로케이션은 Amazon CloudFront에서 짧은 지연 시간과 빠른 전송 속도로 전 세계 고객에게 데이터, 동영상, 애플리케이션 및 API를 안전하게 전송하는데 사용됨<br />

### 컴퓨팅 서비스

**AWS 핵심 기술**<br />
AWS는 컴퓨팅, 스토리지, 데이터베이스, 보안 관리 및 네트워킹 범주에 걸쳐 다양한 기본 인프라 서비스를 제공<br />
AWS 고객은 이러한 제품 및 서비스를 사용하여 웹 및 모바일 애플리케이션, 데이터 처리 및 웨어하우징, 스토리지, 아카이브 등 다양한 워크로드 및 애플리케이션을 구동함

**컴퓨팅 서비스**<br />
컴퓨팅 서비스를 사용하여 AWS 클라우드에서 애플리케이션과 워크로드를 개발, 배포, 실행 및 조정할 수 있음
- Amazon Elastic Compute Cloud(Amazon EC2)는 클라우드에서 안전하고 크기 조정이 가능한 컴퓨팅 용량을 제공함
    - 컴퓨팅 용량은 전통적으로 온프레미스 환경의 물리적 서버 또는 가성 서버에 의해 제공되었던 기능
- Amazon EC2 Auto Scaling을 사용하여 고객은 배포 환경에서 인스턴스 수가 자동으로 증가 또는 감소하도록 트리거를 설정하여 애플리케이션의 상태 및 가용성을 유지
- 고객은 수신되는 애플리케이션 트래픽을 Amazon EC2 인스턴스, 컨테이너, IP 주소, Lambda 함수 등 여러 대상으로 자동으로 분산하는 Elastic Load Balancing(ELB)라고 하는 로드 밸런싱 서비스를 사용
- AWS Lambda는 서버를 프로비저닝하거나 관리할 필요 없이 코드를 실행할 수 있는 컴퓨팅 서비스 -> 서버리스 서비스
    - 이미지 업로드, 인앱 활동, 웹사이트 클릭, 연결된 디바이스에서의 출력과 같은 이벤트가 발생한 후 몇 밀리 초 내에 코드를 실행
- Amazon EC Container Service(ECS)는 Docker 컨테이너를 지원하는 확장성이 뛰어난 고성능 컨테이너 관리 서비스
    - 고객은 이서비를 사용하여 Amazon EC2 인스턴스의 관리형 클러스터에서 애플리케이션을 손쉽게 실행

**Amazon EC2의 이점**<br />
1. 탄력성
    - Amazon EC2를 사용하면 고객은 몇 분 내에 용량을 능리거나 줄일 수 있음.
    - 이후 학습할 Amazon EC2 Auto Scaling과 결합하면 고객은 Amazon EC2 플릿의 가용성을 유지하고 필요에 따라 자동으로 확장 및 축소하여 성능을 극대화하고 비용을 최소화할 수 있음
1. 완전한 제어
    - 고객은 루트 액세스 및 상호 작용 기능을 포함하여 다른 머신에서와 같이 인스턴스를 완전히 제어할 수 있음
1. 유연한 클라우드 호스팅 서비스
    - 고객은 여러 인스턴스 유형, 운영체제 및 소프트웨어 패키지를 선택 가능
1. 통합
    - Amazon EC2는 대부분의 AWS 서비스와 통합되어 있어 컴퓨팅, 쿼리 처리 및 다양한 애플리케이션 간 클라우드 스토리지에 대해 완전하고 안전한 솔루션을 제공
1. 안정성
    - Amazon EC2는 교체 인스턴스를 빠르고 예측 가능하게 실행할 수 있는 매우 안정적인 환경을 제공
    - Amazon의 입증된 네트워크 인프라와 데이터 센터 내에서 실행됨
1. 보안성
1. 저렴한 비용
1. 손쉬운 시작

**Amazon EC2 선택: 인스턴스 유형**<br />
AmazonEC2는 다양한 사용 사례에 맞게 최적화된 다양한 하드웨어 및 소프트웨어 구성을 제공<br />
인스턴스 유형은 CPU, 메모리, 스토리지, 및 네트워킹 용량의 다양한 조합으로 구성<br />
인스턴스는 먼저 다음과 같은 일반 범주로 나뉨
- 범용 인스턴스는 균형있는 컴퓨팅, 메모리, 및 네트워킹 리소스를 제공
- 컴퓨팅 최적화 인스턴스는 고성능 프로세서를 활용하는 컴퓨팅 집약적인 애플리케이션에 적합
- 메모리 최적화 인스턴스는 메모리에서 대규모 데이터 세트를 처리하는 워크로드를 위한 빠른 성능을 제공
- 가속화 컴퓨팅 인스턴스는 하드웨어 액셀러레이터 또는 코프로세서를 사용하여 부동 소수점 수 계산이나 그래픽 처리, 데이터 패턴 일치 등의 기능을 효율적으로 수행
- 스토리지 최적화 인스턴스는 로컬 스토리지의 매우 큰 데이터 세트에 대한 순차적 읽기 및 쓰기 액세스가 많이 필요한 워크로드를 위해 설계
    - 애플리케이션에 대해 지연 시간이 짧은, 수만 단위의 무작위 IOPS를 지원하도록 최적화

**Amazon Machine Image(AMI)**<br />
인스턴스 유형은 인스턴스의 하드웨어 기능을 나타냄<br />
Amazon Machine Image(AMI)는 인스턴스의 초기 소프트웨어 구성을 나타냄<br />
인스턴스를 처음 시작할 때 고객은 인스턴스 시작 시의 소스 AMI를 지정
동일한 구성의 인스턴스가 여러 개 필요할 때는 한 AMI에서 여러 인스턴스를 시작할 수 있음<br />
고객은 서로 다른 구성의 인스턴스가 필요할 때 다양한 AMI를 사용하여 인스턴스 시작 가능<br />
고객은 AWS, AWS Marketplace, 사용자 커뮤니티에서 제공하는 AMI를 선택할 수 있음<br />
고객은 사용자 지정 AMI를 생성하고 유지 관리할 수도 있음

**규모 조정이 중요한 이유**<br />
기존 데이터 센터 환경에서는 시스템의 확장성이 물리적 하드웨어에 따라 한정됨<br />
클라우드에서는 컴퓨팅 파워가 프로그래밍 방식의 리소스이므로 규모 조정 문제에 유연하게 접근 가능
- 피크 기간 전에 새 인스턴스 시작
- 모니터링을 사용하여 프로그래밍 방식으로 확장
- 자동 축소
- 필요할 때 필요한 리소스에 대해 비용 지불

**Amazon EC2 Auto Scaling**<br />
애플리케이션 가용성을 유지하고 사용자가 정의한 조건에 따라 EC2 인스턴스를 자동으로 추가하거나 제거할 수 있음
- 실행 중인 인스턴스의 상태 모니터링
- 손상된 인스턴스 자동 교체
- 여러 가용 영역에서 용량 밸런싱
- 동적이고 예측 가능한 규모 조정

**Amazon Elastic Load Balancing**<br />
여러 EC2 인스턴스 간에 트래픽을 자동으로 분산시켜 내결함성 및 가용성을 높임<br />
클라이언트에서 오는 트래픽을 어용하고, 하나 이상의 가용 영역에서 등록된 대상으로 요청을 라우팅함
ELB는 애플리케이션의 가용성과 내결함성을 높임<br />
고객은 로드 밸런서가 정상적인 대상에만 요청을 보내도록 컴퓨팅 리소스 상태를 모니터링 하는 상태 확인 구성<br />
컴퓨팅 리소스가 주요 작업에 집중할 수 있도록 암호화 및 복호화 작업을 로스 밸런서로 오프로드<br />
ELB는 세 가지 유형으로 로드 밸런서 지원
- Application Load Balancer: HTTP 및 HTTPS 트래픽을 사용하는 웹 애플리캐이션에 맞춰 설계 (애플리케이션 계층에서 작동)
- Network Load Balancer: 갑작스러운 트래픽 급증에 대한 복원력이 뛰어남 (네트워크 계층에서 작동)
- Classic Load Balancer: Amazon EC2 클래식 인스턴스용으로 설계된 이전 세대 로드 밸런서.

### 스토리지 서비스

**AWS 스토리지 서비스**<br />
AWS는 확장 가능한 내구력 있는 객체 스토어인 Amazon Simple Storage Service(S3)와 Elastic Block Storage(EBS)라는 블록 스토어 등 여러 스토리지 서비스를 제공
- Amazon EBS
    - Amazon EC2 인스턴스용 하드 드라이브와 같음
    - 영구적인 블록 수준 스토리지 볼륨을 제공
- Amazon S3
    - 개발자가 웹 규모의 컴퓨팅 작업을 보다 쉽게 할 수 있도록 설계된 인터넷용 스토리지 서비스
    - 언제 어디서나 웹에서 용량과 관계없이 데이터를 저장하고 검색하는 데 사용할 수 있는 단순한 웹 서비스 인터페이스를 제공
- Amazon S3 Glacier
    - 보안성과 내구성이 우수한 데이터 보관 및 백업용 스토리지를 제공하는 스토리지 서비스
    - 간편하고 비용 효율적으로 수개월에서 수십년 동안 데이터를 보관할 수 있음
- AWS Storage Gateway
    - 온프레미스 IT 환경과 AWS 스토리지 인프라를 원활하고 안전하게 통합
- Amazon Elastic File System
    - Amazon EC2 인스턴스용 파일 스토리지 서비스
    - 사용이 간편하며 용량은 탄력적으로 운영됨

**Amazon Elastic Block Storage(EBS)**<br />
Amazon EC2 인스턴스에서 사용할 수 있는 내구성이 뛰어난 블록 수준 스토리지
- 인스턴스와 독립적으로 유지
- 물리적 하드 드라이브처럼 사용
- 장애로 인한 데이터 손실을 방지하기 위해 자동으로 복제
- 동일한 AZ의 인스턴스에 연결됨
- 하나의 EBS 볼륨당 한 번에 하나의 EC2 인스턴스에만 연결 가능
- 단일 인스턴스에 여러 EBS 볼륨 연결
- EC2 인스턴스 종료 후 EBS 볼륨에 데이터 보존 가능
- 볼륨의 지정 시간 스냅샷을 생성해 Amazon S3에 저장 가능

**Amazon Simple Storage Service(Amazon S3**<br />
데이터를 버킷이라는 리소스 내 객체 단위로 저장
- 객체의 크기는 최대 5TB이지만 사용 가능한 총 스토리지, 객체의 양에는 제한이 없음
Amazon S3는 99.999999999%의 내구성과 99.99%의 가용성을 제공<br />
일반적인 S3 사용 사례
- 백업 및 스토리지
- 애플리케이션 호스팅: 웹 애플리케이션 배포, 설치, 관리하는 서비스
- 미디어 호스팅: 미디어 파일의 업로드 및 다운로드를 호스팅하는 중복 방식의 확장 가능하고 가용성 높은 인프라 구축
- 소프트웨어 전송: 고객이 다운로드할 수 있는 소프트웨어 애플리케이션을 호스팅

**Amazon S3의 스토리지 클래스**<br />
고객의 데이터의 유형, 볼륨 및 액세스 특성에 적합한 스토리지 클래스 활용 가능
- AWS S3 Standard
    - 자주 액세스 하는 데이터를 위한 서비스
    - 짧은 지연 시간과 높은 처리량을 제공
- Amazon S3 Standard - Infrequent Access(S3 Standard-IA)
    - 액세스 빈도가 낮지만 필요할 때 빠르게 액세스해야 하는 데이터를 위한 스토리지 클래스
    - 장기 스토리지, 백업 및 재해 복구 파일용 데이터 스토어에 적합
- Amazon One Zone - Infrequent Access(S3 One Zone-IA)
    - 액세스 빈도가 낮은 데이터용으로 사용
    - 다른 S3 스토리지 클래스는 최소 3개의 가용 영역에 데이터를 저장하는 반면, S3 One Zone-IA는 단일 AZ에 데이터를 저장
        - S3 Standard-IA보다 20@ 저렴함
    - S3 Standard, S3 Standard-IA 스토리지와 같은 가용성 및 복원력이 필요하지 않는 고객에게 적합
    - 온프레미스 데이터 또는 쉽게 다시 생성할 수 있는 보조 백업 복사본을 저장하는 경우 좋은 옵션
- Amazon S3 Glacier(S3 Glacier), Amazon Glacier Deep Archive(S3 Glacier Deep Archive)
    - 데이터 아카이브를 위한 안전하고 내구성이 뛰어나며 비용이 저렴한 스토리지 클래스
- Amazon S3 Intelligent-Tiering
    - 고객이 구성한 두 액세스 티어 간에 데이터를 이동
### 데이터베이스 서비스

**데이터베이스 서비스**<br />
AWS 데이터베이스 서비스는 백업, 패치 관리, 복제와 같은 시간 소모적인 데이터베이스 관리 작업을 처리해주기 때문에, 고객은 애플리케이션 개발이나 데이터베이스 개선 등 더 가치있는 일에 집중할 수 있음
- Amazon Relational Database Service(Amazon RDS)
    - Amazon Aurora, PostgreSQL, MySQL, MariaDB, Oracle Database, SQL Server 등 주로 사용되는 6개 데이터베이스 엔징 중에서 선택 가능한 관리형 데이터베이스 서비스
    - 고객은 클라우드에 비용 효율적이고 크기 조정 가능한 용량을 제공하는 관계형 데이터베이스를 설치, 운영 및 조정 가능
    - 자동화된 백업, 데이터베이스 스냅샷, 자동 하스팅 교체를 비롯한 중요한 프로덕션 데이터베이스의 안정성을 강화하는 여러가지 기능이 있음
- Amazon DynamoDB
    - 어떤 규모에서도 10ms 미만의 성능을 제공하는 NoSQL 데이터베이스
    - 간단하게 데이터베이스 테이블을 시작
    - 가동 중지나 성능 저하 없이 용량을 조정
    - 리소스 사용륭과 성능 지표를 시각적으로 관리할 수 있음
- Amazon ElastiCache
    - 클라우드에서 인 메모리 캐시를 손쉽게 배포, 운영 및 조정할 수 있게 해주는 캐싱 서비스
    - 빠른 관리형 인 메모리 캐시 시스템에서 정보를 검색할 수 있는 기능을 갖춰 웹 애플리케이션의 성능을 향상

**EC2 호스팅 vs. AWS 데이터베이스 서비스**<br />
- Amazon 데이터베이스 서비스
    - 데이터베이스 프로지버닝, 백업 및 복구, 보안 패치, 버전 관리 같은 획일적인 작업을 AWS가 대신 수행
    - 즉각적이고 동기식인 다중 AZ 배포를 통한 고가용성 데이터베이스 솔루션 이용 가능
    - Orable Database Standard Edition One/Two의 고가용성 예비 인스턴스에 대한 동기식 복제가 가능
    - 자동 백업/복구
    - 워크로드 패턴을 바탕으로 인스턴스 유형의 규모를 확장하거나 축소할 수 있음
- Amazon EC2의 데이터베이스
    - Sys/System 사용자 액세스, 운영체제 수준의 액세스를 포함한 데이터베이스 완벽 제어

### 네트워킹

**네트워킹 서비스**<br />
- Amazon Virtual Private Cloud(VPC)
    - AWS 클라우드에서 가상 네트워크를 구축 가능
    - 격리된 프라이빗 네트어크 영역을 프로비저닝하고 여기에 리소스를 배포할 수 있음
    - 보안 그룹과 네트워크 액세스 제어 목록과 같은 고급 보안 기능 제공
- Amazon Route 53
    - 확장성이 뛰어난 DNS 서비스

**Amazon Virtual Private Cloud(VPC)**<br />
AWS 리소스를 고객이 정의한 가상 네트워크에서 실행 가능<br />
VPC를 통해 가상 네트워킹 환경을 완벽하게 제어
- 고객은 IP 주소 범위 선택, 서브넷 생성, 라우팅 테이블 및 네트워크 게이트웨이 구성 등

**VPC 보안 유지**<br />
VPC 내에서 각 서브넷을 고사는 네트워크 트래픽은 네트워크 ACL(액세스 제어 목록)이라는 다른 선택적 보안 계층을 사용하여 허용/거부될 수 있음<br />
VPC 내에서 실행되는 Amazon EC2 인스턴스의 보안을 위해 인스턴스 수준에서 가상 방화벽 역할을 하는 보안 그룹을 생성할 수 있음
- 인스턴스에는 최대 5개의 보안 그룹이 할당될 수 있음
    - VPC에 있는 서브넷의 각 인스턴스를 서로 다른 보안 그룹 세트에 할당 가능
    - VPC는 기본 보안 그룹과 함께 자동으로 제공
VPC 흐름 로그는 VPC, 서브넷 또는 네트워크 인터베이스에 대한 네트워크 흐름 정보를 캡처하고 Amazon CloudWatch Logs에 저장<br />
호스트 기반 방화벽은 고객이 사용할 수 있는 운영 체제 방화벽
- iptable, Windows 방화벽과 같은 내장 OS 또는 심층 패킷 검사나 네트워크 위협 방지와 같은 추가 기능을 제공하는 타사 소프트웨어가 포함
호스트 기반 보안 소프트웨어는 전체 소프트웨어 플릿에 네트워크 패킷 검사가 분산되므로 고도로 분산되고 확장 가능한 애플리케이션 아키텍처에 효과적

### 보안, 자격 증명 및 규정 준수

**보안 자격 증명 및 규정 준수 서비스**<br />
AWS는 고객이 데이터를 최대한 안전하게 유지할 수 있도록 보안, 자격 증명 및 규정 준수 서비스를 제공
- AWS IAM, 공동 책임 모델, 클라우드 규정 준수

**Amazon Identity and Access Management(Amazon IAM)**<br />
AWS 서비스와 리소스에 대한 액세스를 안전하게 관리할 수 있음<br />
AWS 사용자 및 그룹을 만들고 관리하며 AWS 리소스에 대해 액세스를 허용/거부할 수 있음
- 특정 사용자를 특정 리소스나 작업으로 제한하거나 특정 시간대에 제한 하는 등 AWS 리소스에 대한 세밀한 액세스 제어 가능
- 권한이 높은 사용자용 멀티 팩터 인증
- 사용중인 서비스에만 액세스할 수 있도록 정책을 쉽게 식별, 구체화하는 등의 액세스 분석 기능
- 기업 디렉터리와의 통합을 통해 Microsoft Active Directory와 같은 자격 증명 시스템의 사용자에게 연동 액세스 허용

**AWS의 클라우드 보안**<br />
- AWS 데이터 센터 및 네트워크 아키텍처의 이점을 상속
- 온프레미스 데이터 센터와 유사하지만 시설 및하드웨어 유지 관리가 필요없음
- 손쉽게 자동화 가능
- AWS의 모든 모범 사례를 상속받아 유연하고 민첩한 보안 제어가 가능

**AWS 공동 책임 모델**<br />
보안 및 규정 준수는 공동의 책임
- AWS는 **클라우드 자체의 보안**에 대한 책임이 있음
    - AWS 클라우드가 구현하고 운영하는 보안 조치를 책임짐
    - AWS 클라우드의 모든 서비스를 실행하는 글로벌 인프라를 보호할 책임을 가짐
- 고객은 **클라우드 내부의 보안**에 대한 책임이 있음
    - AWS 서비스를 이용하는 고객 콘텐츠 및 애플리케이션의 보안과 관련하여 고객이 시행, 운영하는 보안 조치
    - 자신의 콘텐츠, 플랫폼, 애플리케이션, 시스템 및 네트워크를 보호하기 위한 보안 조치를 구현하고 직접 관리함
- 컨설팅 파트너는 클라우드에서 고객의 책임 영역을 해결하는 솔루션을 개발함으로써 고객을 위한 품질을 높일 수 있음

**AWS 클라우드 규정 준수**<br />
AWS 환경은 지역 및 업종에 따른 인증 기관의 인증과 더불어 지속적으로 감사를 받음<br />
AWS 환경에서 고객은 자산 인벤토리를 위한 자동화 도구와 권한 액세스 보고를 활용할 수 있음<br />
AWS는 고객과 관련된 보안 및 제어 환경을 알려야 함
- 산업 인증 및 독립적인 타사 인증 획득
- 백서와 웹사이트 콘텐츠를 통한 AWS 보안 및 제어 관행에 대한 정보 공개
- NDA에 따라 AWS 고객에게 인증서, 보고서, 기타 문서 직접 제공
AWS는 외부 인증 기관 및 독립 감사자와 협력하여 고객에게 AWS에서 수립, 운영하는 정책, 프로세스 및 컨트롤에 대한 정보 제공<br />
규정 준수 인증 및 검증은 독립적인 외부 감사 기관이 평가하여 인증, 감사 보고 또는 규정 준수 검증의 형태로 결과가 나타남<br />
AWS 고객은 관련 규정, 법률을 준수할 책임이 있음

### 관리 인터페이스

AWS 사용자는 세 가지 방법으로 플랫폼에서 리소스를 생성하고 관리할 수 있음<br />
세 가지 옵션 모두 AWS의 기초 역할을 하는 공통적인 REST 유형의 API에 기반함

- AWS SDK: 널리 사용되는 다양한 프로그래밍 언어로 서비스에 액세스

## M3. 서비스부터 솔루션까지

**AWS 솔루션**<br />
- 엔터프라이즈 애플리케이션: 대기업의 고유한 보안, 규정 준수, 프라이버시 및 거버넌스 요구 사항에 맞춰 특별히 설계된 완성된 서비스 세트를 사용해 구축
- 스토리지: 페타바이트 규모의 백업, 재해 복구 및 데이터 아카이브를 위한 안정적이고 비용 효율적인 옵션
- 서버리스 컴퓨팅: 서버에 대한 고민 없이 애플리케이션과 서비스를 구축하고 실행
- Windows 워크로드: Microsoft 애플리케이션을 위한 유연하고 확장 가능한 컴퓨팅 용량을 제공. Windows 워크로드를 손쉽게 관리 및 보호
- 컨테이너: 경량 애플리케이션을 패키징 및 배포하고 애플리케이션을 어디에서든 쉽게 실행하고 확장할 수 있도록 일관된 이동식 소프트웨어 환경을 제공
- 기계학습: 강력한 서비스와 플랫폼으로 구축하고, 어디에서나 광범위한 기계학습 프레임워크를 제공
- 사물 인터넷: 물리적 세계를 클라우드에 연결하는 유비쿼트스 디바이스 시스템
- 분석 및 데이터 레이크: 하나의 중앙 집중식 리포지토리에서 모든 데이터를 안전하게 저장, 분류 및 분석

### 고객의 당면 과제 해결

**AnyCompany의 현재 구성**<br />
- Java로 개발된 모놀리식 전자 상거래 웹 애플리케이션
- 별도의 웹 티어와 데이터베이스 티어를 의미하는 2티어 솔루션
- 웹 프론트엔드와 애플리케이션은 로드 밸런싱된 웹/앱 서버 쌍에 배포
    - 웹/앱 서버는 Ubuntu Linux 배포판에서 실행중인 Apache Web SErver를 실행
    - 고객이 액세스 하는 퍼블릭 서브넷에서 개방된 방화벽을 통해 80, 443이 실행 중
- 데이터베이스는 이중화 없이 단일 서버에서 실행되는 MySQL
    - 방화벽 뒤에 있는 프라이빗 서브넷에 위치
    - 하드 코딩된 IP 주소 규칙을 통해 웹/앱 서버에서 3306의 트래픽만 허용
    - 데이터베이스는 주간 전체 백업과 상용 테이프 백업 소프트웨어를 사용해 매일 밤 증분 백업

### 마이그레이션 전략

**6R**<br />
- 리호스팅(Rehost), 리플랫폼(Replatform), 리팩터링(Refactor), 사용 중지(Retire), 유지(Retain), 재구매(Repurchase)

**리호스팅(Rehost): 리프트 앤 시프트**<br />
- 온프레미스에서 호스팅 되는 애플리케이션에서 마이그래이션하는 가장 일반적인 방식
    - 온프레미스 데이터 센터에서 AWS 클라우드로 애플리케이션 리소스를 이전하는 것과 비슷함
- 서버는 EC2 인스턴스로 마이그레이션
- 네트워크 구성은 VPC, 서브넷, 보안 그룹, 인터넷 및 VPN 게이트웨이 등을 사용하여 다시 생성
    - AWS 클라우드에서 온프레미스 네트워크를 재현
- AWS VM Import/Export와 같은 도구를 사용하여 자동화할 수 있음
- 마이그레이션 이후 더욱 쉽게 애플리케이션을 최적화하고 재설계할 수 있음

**리플랫폼(Replatform): 리프트 앤 시프트 수정**<br />
- 애플리케이션의 핵심 아키텍처가 변경되지 않는다는 점에서는 리호스팅과 유사
- 맞춤형 AWS 클라우드 최적화
    - RDS와 같은 관리형 솔루션으로 전환하거나 Amazon Elastic Beanstalk 같은 완전관리형 플랫폼으로 애플리케이션을 마이그레이션
- 아키텍처 모범 사례를 사용한 인프라 최적화

**리팩터링(Refactor): 현대화**<br />
- 클라우드 네이티브 기능을 사용하여 애플리케이션의 설계 및 개발 방식을 재설계
- 비즈니스 요구사항으로 인해 다른 방법으로는 기존 환경의 애플리케이션에서 실현하기 까다로운 기능 추가, 확장 또는 성능 개선의 필요성이 클 때 활용됨
    - 데이터베이스 구조를 EC2 호스팅 데이터베이스나 표준 RDS 데이터베이스로 변경
    - 모놀리식 애플리케이션 아키텍처를 서비스 중심 설계로 변경

**기타 전략**<br />
- 사용 중지(Retire)
    - 유용하지 않은 애플리케이션 종료
    - 지출, 관리 및 보안 감소
- 유지/재검토 (Retain/Revisit)
    - 특정 애플리케이션을 온프레미스로 유지
- 재구매 (Repurchase)
    - SaaS(Software as a Service)로 워크플로 이전

**Maria의 마이그레이션 전략**<br />
- 리플랫폼(Replatform)
    - 일부 기능을 깄는 그대로 마이그래이션
    - 아키텍처의 다른 구성요소를 최적화하여 복원력을 개선

### 클라우드 아키텍처 모범 사례

1. 장애 대비 설계로 인한 무장애
    - 단일 장애 발생 지점 제거
    - 다중 인스턴스
    - 복수 가용 영역
    - 단일 서버를 여러 계층화된 애플리케이션으로 분리
    - Amazon RDS의 경우 다중 AZ 기능 사용
1. 모든 계층에서 보안을 고려
    - 저장 및 전송 중 데이터 암호화
    - IAM에서 최소 권한의 원칙 적용
    - 보안 그룹과 NACL(네트워크 액세스 제어 목록)을 모두 구현
    - 고급 보안 기능 및 서비스 고려
1. 다양한 스토리지 옵션의 활용
    - 정적 웹 자산을 Amazon S3로 이동
    - Amazon CloudFront를 사용하여 전역적으로 서비스 제공
    - DynamoDB에 서비스 제공
    - 호스트와 데이터베이스 간에 ElastiCache 사용
1. 탄력성 구현
    - Auto Scaling 정책 구현
    - 재부팅 및 재시작 복원력 설계
    - Amazon S3 및 Amazon DynamoDB와 같은 관리형 서비스 활용
1. 병렬적 사고
    - 수직이 아닌 수평 확장
    - 세션/상태에서 컴퓨팅 분리
    - Elastic Load Balancing 사용
    - 적절한 인프라 규모 조정
1. 소결합에 의한 융통성
    - 순서가 지정된 단일 워크플로 대신 여러 대기열 사용
    - Amazon Simple Queue Service(SQS) 및 Simple Notification Service(SNS) 사용
    - 기존 서비스 활용
1. 제약을 걱정할 필요가 없음
    - 기존 제약 조건 재고
    - 추가 RAM 설치가 아닌 다수의 상용 인스턴스에 로드 분산
    - 기존 솔루션은 IOPS를 높이기 위해 관계형 스키마를 재작업함.
        - AWS에서는 로드를 분산하여 수평적으로 확장하는 것을 고려
        - 데이터베이스의 읽기 전용 복제본을 생성해 데이터베이스 읽기를 쓰기에서 구분하도록 애플리케이션을 수정
    - 하드웨어 장애 또는 구성 손상에 대해 고려할 필요 없음
### Well-Architected 프레임워크

**Well-Architected 프레임워크란?**<br />
- 모범 사례에 따라 솔루션을 설계하는 데 도움이 되는 중요한 리소스
- 최대한 안전하고 성능이 뛰어나며 복원력이 있고 효율적인 인프라를 클라우드 네이티브 방식으로 구축하는 것을 돕도록 만들어짐
    - AWS 내 수천 개의 고객 아키텍처를 검토하여 구축
- 5개 핵심 요소에 따라 아키텍처를 평가하고 설계를 구현하기 위한 체계적인 접근 방식 제공
- AWS 콘솔의 Well-Architected 도구를 통해 고객의 워크로드 상태를 검토하고 최신 AWS 아키텍처 모범 사례와 비교할 수 있음

**다섯가지 핵심 요소**<br />

- 보안

    - 주요 주제: 데이터의 기밀성 및 무결성, 권한 관리를 통한 사용자 작업 식별 및 관리, 시스템 보호와 보안 이벤트 탐지를 위한 제어 설정 등
- 안정성

- AWS 솔루션: 고객의 일반적인 문제를 해결하고 더욱 빠르게 구축할 수 있도록 설계된, 검증된 기술적 참조 구현
- AWS 솔루션 공간: AWS 컴피턴시 파트너 솔루션을 강조하여 배포 관련 도움이 필요한 고객에게 AWS 솔루션을 제공

**AWS Quick Start**<br />

- 각 Quick Start에는 배포를 자동화하는 AWS CloudFormation 템플릿과 아키텍처를 설명하고 단계별 배포 지침을 제공하는 안내서가 포함

### 솔루션 제안

**AnyCompany의 현재 아키텍처**<br />
- 2티어 솔루션
    - 웹
    - 데이터베이스
- 액세스
    - 웹티어: 웹 포트의 고객
    - 데이터베이스 티어: 데이터베이스 포트의 웹 티어
- 야간에 데이터베이스를 테이프에 백업

**AnyCompany의 솔루션 설계**<br />
- 기존의 2티어 애플리케이션은 3티어 애플리케이션으로 진화
- 데이터 베이스 기능이 MySQL 엔진을 사용하여 Amazon RDS로 마이그레이션됨
    - 데이터베이스 관리 오버헤드가 크게 감소하고 자동 장애 조치 및 S3로의 백업이 제공
- 베스천 호스트 추가
    - 베스천 호스트는 프로덕션 호스트 관리 전용 호스트
    - 베스천 호스트는 모든 인스턴스의 단일 관리 지점이 됨
- 조정 문제를 해결하기 위해 웹 티어와 애플리케이션 티어 모두 Elastic Load Balancing과 Auto Scaling을 사용
- S3는 정적 웹 콘텐츠를 호스팅하기 위해 배포
    - 정적 웹 콘텐츠는 Amazon CloudFront를 통해 서비스되므로 지리적 위치에서 더욱 강력한 고객 경험을 제공
- 애플리케이션 가용성은 Auto Scaling과 2개의 가용 영역을 사용하면서 달성
- 보안
    - 애플리케이션에서는 웹 호스트만 퍼블릭 서브넷에 존재
    - 애플리케이션 및 RDS 인스턴스는 프라이빗 서브넷에 존재
    - 보안 그룹
        - 웹 호스트는 웹 요청에 대해 포트 80 및 443에서 인터넷 트래픽을 허용
        - 애플리케이션 호스트는 웹 프록시로부터의 애플리케이션 포트 트래픽만 연결 허용
        - RDS 인스턴스는 애플리케이션 호스트의 포트 3306에서의 데이터베이스 연결만 허용
        - 베스천 호스트는 AnyCompany 내부 네트워크에서만 관리 연결을 허용하고 다른 모든 호스트는 베스천 호스트에 대한 관리 연결만 허용
        - 서브넷으로의 트래픽을 필터링하는 네트워크 액세스 제어 목록(NACL) 규칙(네트워크 보안의 추가 계층임)
    - S3 버킷은 보안 기능이 활성화 된 상태로 구현
    - 최소 권한 원칙과 함께 연결된 그룹 및 역할이 구성된 표준 IAM 정책도 활성화된 상태로 구현
- 추가로 모니터링 및 로깅, 중요 이벤트에 대한 알림 구성

    - 개인적인 경험, 조사 또는 일화로 인한 이의 발생 가능
    - 대부분의 의사 결정이 논리적이기 보다는 정서적으로 이루어지는 경우가 많음
    - 논리적 구성 요소 이전에 감정적 구성 요소에 대처해야 함
- 먼저 고객의 우려 사항을 인정하며 고객의 관점 파악
- 문제를 핵심 우려 사항으로 압축, 분리
    - 문제가 간결할수록 더 쉽게 해결 가능
- 계속 진행하면서 해결을 위한 조치 시행

**이의 대처 모범 사례**<br />
- 데이터 기반 접근 방식: 지표와 데이터는 주장에 대한 신뢰성을 높임
- 사례 연구 활용: 유사한 솔루션을 성공적으로 구현한 사례는 고객에게 설득력과 위안을 제공
- 심층 분석: 우려의 근본 윈인과 이의의 핵심 파악
- 근거 준비: 확신을 가지고 답하고, 근거가 되는 자료를 활용하여 포지셔닝을 강화
    - 올바른 정보를 즉시 확인할 수 없는 경우, 빠르게 정보를 제공해주겠다고 고객을 설득
- 상황 유지: 질문이나 이의 제기로 대화가 다른 곳으로 흘러서는 안됨

- 영업 주기 및 고객 채택 가속화

## M5. 미리 내다보기

### 솔루션 구현

**마이그레이션 프로세스**<br />
각 단계는 성공적인 마이그레이션의 공통 구성 요소이지만 개별 단계가 아닌 반복적인 프로세스<br />
AWS는 4단계 마이그레이션 프로세스의 각 단계를 간소화하고 가속화할 수 있는 포괄적인 AWS 마이그레이션 도구 포트폴리오를 제공
- 평가
    - 준비 상태 파악
    - 잠재적인 비즈니스 결과 파악
- 준비 및 계획
    - 환경 분석
    - 상호 의존성 맵 작성
    - 마이그레이션 전략 결정
    - Well-Architected Landing Zone 결정
        - 각 애플리케이션의 대한 우선 순위가 포함된 상세한 마이그레이션 계획 수립
- 마이그레이션
    - 포트폴리오 수준에서 개별 애플리케이션으로 초점이 전환
    - 각 애플리케이션의 설계, 마이그레이션 및 검증
    - 자동/수동 마이그레이션
    - 데이터 마이그레이션
- 운영 및 최적화
    - 클라우드에서 워크로드를 효율적으로 운영, 관리 및 최적화

**최소 기능 제품(MVP)**<br />
고객을 위한 새로운 솔루션을 구현할 때 최소 기능 제품(Minimum Viable Product, MVP)의 개념을 따를 수 있음
- 고객이 긴 프로세스의 끝이 아닌 처음부터 진행 상황을 볼 수 있도록 하는 좋은 방법
- 최소 기능 제품은 초기 도입시 고객의 요구 사항을 충족하기에 충분한 기능을 갖춘 기능적 제품/솔루션
    - 향후 제품/솔루션 개발을 위한 피드백의 여지가 있음
- 제공 주기 동안 고객이 MVP를 염두에 두고 새로운 기능에 대한 아이디어를 정리하는 것이 좋음
    - 제품 개발 주기가 끝날 때만 가치가 있는 '빅뱅' 방식의 솔루션을 피할 것
- 기본적인 것으로 시작하여 솔루션이 복잡해지는 과정 속에서 피드백을 수집

**MVP 및 결과 제공**<br />
PoC와 MVP의 차이점
- POC: 솔루션의 빠른 데모
- MVP: 프로덕션 수준

**마이그레이션 전략**<br />
- 리호스팅(리프트 앤 시프트)은 전적으로 수동으로 수행하거나, 대부분 자동화할 수 있음
    - 수동 작업: 애플리케이션 구성 요소를 수동으로 설치, 구성 및 배포
        - 완료 후 AWS에서 검증 -> 프로덕션으로 전환
    - 자동화: 마이그레이션 도구는 위의 단계 중 일부 또는 전부를 제거할 수 있음
- 리플랫폼(리프트 앤 시프트)
    - 첫 번째 단계는 애플리케이션의 새 대상 플랫폼을 결정하는것
    - 그 다음 마이그레이션을 준비하기 위해 현재 인프라를 약간 수정(할 수 있음)
    - 마이그레이션 도구를 사용하요 보다 원활한 전환 수행 가능
- 리팩터링
    - 먼저 대상 애플리케이션 및 인프라에 맞게 애플리케이션 재설계
    - 대상 아키텍처에 대한 코드 개발
    - 전체 애플리케이션 수명 주기 관리(ALM) 및 소프트웨어 개발 수명 주기 완료
    - 대상 애플리케이션을 완전히 통합

**AnyCompany의 마이그레이션 - 리플랫폼**<br />
핵심 기능과 운영은 동일<br />
비즈니 요구 사항 및 목표에 대한 일부 세부 사항을 충족하기 위해 일부 수정 사항 적용<br />
- 먼저 VPC 및 서브넷 생성하여 네트워크 인프라 생성
    - 포트 보안은 보안 그룹 및 NACL을 통해 추가
    - 사이트에 대한 액세스는 NAT 및 인터넷 게이트웨이에 의해 제공
- 핵심 애플리케이션 기능은 리플랫폼
    - 웹 앱 서버는 별토의 웹 호스트와 애플리케이션 호스트 티어로 분할
    - 세 번째로 베스천 호스트 추가
    - 이 템플릿은 Auto Scaling 및 로드 밸런싱에 사용되는 AMI 템플릿으로 신중하게 구성
    - Amazon RDS에서 MySQL 호스트의 리플랫폼 실시
- Auto Scaling 그룹 및 Application Load Balancer 생성 및 구성을 통해 로드 밸런싱 및 조정이 이루어짐
- 마지막으로 S3 버킷 구성
    - 호스트 정적 웹 자산 및 데이터베이스 백업에 사용
- 모든 것이 구성되면 데이터 및 애플리케이션 코드가 수동으로 웹 및 애플리케이션 인스턴스로 마이그레이션
    - MySQL 데이터베이스는 AWS Database Migration Service를 사용하여 RDS로 마이그레이션

**프로덕션**<br />
- 고객이 AWS에서 프로덕션을 시작할 준비가 된 경우 그에 따른 몇 가지 모범 사례가 있음
    - 해당 고객의 AWS 계정 팀에 연락하여 새 프로젝트를 시작할 준비가 되었음을 알려주는 것이 좋음
        - 필요한 경우 솔루션스 아키텍트 또는 기술 지원 관리자(TAM)가 기술 지원을 제공할 수 있음
    - 고객에게 특정 규제 요구 사항이 있는 경우, 솔루션 준수를 위해 활용해야 하는 구현 체크리스트가 있는지 확인
        - AWS 공동 책임 모델에 따라 고객은 솔루션의 특정 규정을 준수할 책임이 있음
        - AWS에서 지침을 제공할 수 있음
    - 고객의 AWS 지원 수준에 따라 달라짐
        - 고객이 엔터프라이즈 지원을 선택할 경우, 사전에 환경을 모니터링하고 최적화를 지원할 수 있는 기술 계정 관리자를 이용할 수 있음

**Well-Architected 검토**<br />
솔루션 구현 후에는 워크로드에 대해 정기적으로 Well-Architected 검토를 실행<br />
AWS Well-Architected Tool은 워크로드 상태를 검토하고 최신 AWS 아키텍처 모범 사례와 비교하는 데 도움이 됨
- 아키텍처 지침을 제공: 클라우드에 맞게 더 나은 워크로드를 구축하는 방법에 대한 단계별 지침과 함께 실행 계획을 제공
- 워크로드를 일관되게 검토: 전반적인 상태를 모니터링하여 잠재적 위험을 이해
- 개선 사항 파악 및 구현

**AWS Well-Architected Tool**<br />
- AWS Well-Architedcted 검토에서는 AWS Management Console에서 제공하는 무료 AWS Well-Architedted Tool을 사용
    - 먼저 문서화할 워크로드 식별
    - 아키텍처에 대한 일련의 질문 응답
    - Well-Architected 프레임워크에서 확립한 다섯 가지 핵심 요소(운영 우수성, 보안, 안정성, 성능 효율성, 비용 최적화)와 비교해 답변 검토
    - 완료 후, 동영상 및 설명서, 워크로드 검토 요약 보고서, 검토 결과를 단일 대시보드에 제공

### 현대화

**현대화를 통한 성장 촉진**<br />
- 새로운 애플리케이션을 구축하고 레거시 솔루션을 폐기할 때, AWS를 사용하여 비용을 절감할 수 있음
- 고객이 클라우드 마이그레이션을 계획하는 경우, 조직은 TCO(Total Cost of Ownership)를 크게 절감하여 서버 및 건물 관리가 아닌 기업의 핵심 미션에 집중할 수 있는 리소스 확보 가능
- AWS의 방대한 규모, 광범위한 서비스, 자동화 수준을 통해 점진적인 절감 달성
- AWS에서 엔터프라이즈 솔루션을 배포하는 경우
    - 값 비싼 레거시 인프라를 폐기하고
    - 비용을 절감하며
    - 자동화를 통해 민첩성을 확보하고
    - 획일적인 작업에 집중하는 대신 혁신을 추진하는 데 필요한 리소스 확보

**아키텍처 현대화**<br />
- 서버리스, 컨테이너, 데이터 레이크 및 분석

**서버리스**<br />
- AWS는 서버리스 애플리케이션을 구축, 실행하는데 사용할 수 있는 일련의 완전 관리형 서비스를 제공
- 서버리스 애플리케이션을 사용하면
    - 컴퓨팅, 데이터베이스, 스토리지, 스트림 처리, 메시지 대기열 등의 백엔드 구성요소를 위한 서버를 프로비저닝, 유지, 관리할 필요가 없음
    - 더는 애플리케이션 내결함성 및 가용성에 신경 쓸 필요가 없음
    - 이를 통해 제품 출시 시간을 단축하고, 동시에 제품 혁신에 집중할 수 있음
- 서버리스용 AWS 완전관리형 서비스
    - 컴퓨팅
        - AWS Lambda
        - Lambda@Edge
        - AWS Fargate
    - 스토리지
        - Amazon Simple Storage Service
        - Amazon Elastic File System
    - 데이터 스토어
        - Amazon DynamoDB
        - Amazon Aurora Serverless
    - API 프록시
        - Amazon API Gateway
    - 애플리케이션 통합
        - Amazon SNS
        - Amazon SQS
        - AWS AppSync
        - Amazon EventBridge
    - 오케스트레이션
        - AWS Step Functions
    - 분석
        - Amazon Kinesis
        - Amazon Athena
    - 개발자 도구
        - 다양한 도구 및 서비스

**컨테이너**<br />
- 컨테이너는 애플리케이션의 코드, 구성 및 종속성을 하나의 객체로 패키징하는 표준화된 방식을 제공
- 컨테이너는 서버에 설치된 운영 체제를 공유하며 리소스가 격리된 프로세스 형태로 실행되므로 환경에 상관 없이 빠른고 안정적이며 일관된 배포를 보장
    - AWS 클라우드는 컨테이너 실행에 최적화된 인프라 리소스를 제공
    - 컨테이너식 애플리케이션을 프로덕션 환경에서 쉽게 구축 및 실행할 수 있게 해주는 일련의 오케스트레이션 서비스를 제공
- 컨테이너의 잠재적 사용 사례
    - 마이크로 서비스
    - 배치성 프로세스
    - 기계 학습
    - 하이브리드 애플리케이션
    - 클라우드로 애플리케이션 마이그레이션
    - 서비스로서의 플랫폼

**분석 및 데이터 레이크**<br />
- 서로 다른 사일로의 데이터에 액세스하여 데이터를 분석하기 어려움
    - 단일 장소, 데이터 레이크에 데이터를 저장하면 이 문제를 해결할 수 있음
- 데이터 레이크를 사용하면 기업이 데이터를 쉽게 읽고 통할력을 확보할 수 있어 운영 최적화가 가능
- AWS는 데이터 레이크 및 분석 솔루션 구축을 위해 데이터의 이동, 저장 및 분석을 지원하는 포괄적인 서비스를 제공
    - 데이터 이전: 온프레미스에서 실시간으로 데이터를 가져옴
    - 데이터 레이크: 기가바이트에서 엑사바이트까지 모든 유형의 데이터를 안전하게 저장
    - 분석 서비스: 가장 광범위한 분석 서비스를 사용하여 데이터를 분석
    - 기계 학습: 미래의 결과를 예측하고 신속한 대응을 위한 조치를 알려줌
- 데이터 레이크를 통한 부가 가치의 예
    - 고객 상호 작용 개선
    - R&D 혁신 옵션 개선
    - 운영 효율성 개선

**AWS 기반 데이터 레이크 분석 솔루션**<br />
- AWS는 데이터 레이크에서 실행되는 가장 광범위하고 비용 효율적인 분석 서비스 세트를 제공
- 대화식 분석, 빅데이터 처리, 데이터 웨어하우징, 실시간 분석, 운영 분석, 대시보드 및 시각화
- 대화식 분석: Amazon Athena
    - 표준 SQL 쿼리를 사용해 S3와 Glacier에 있는 데이터를 직접 간편하게 분석할 수 있음
    - Athena는 서버리스이므로, 설정하거 관리할 필요가 없음
    - 고객은 데이터를 즉시 퀘리하고, 몇 초 내에 결과를 얻고, 실행한 쿼리에 대해서만 비용을 지불
- 빅데이터 처리: Amazon EMR
    - Spark 및 하둡 프레임워크를 사용한 빅데이터 처리
    - 관리형 서비스로서 대량의 데이터를 쉽고 빠르며 비용 효율적으로 처리할 수 있음
- 데이터 웨어하우징: Amazon Redshift
    - 페타바이트의 정형 데이터에 대해 복잡한 분석 퀘리를 실행할 수 있는 기능 제공
    - 불필요한 데이터 이전 없이 S3의 정형 또는 비정형 데이터에 대해 직접 SQL 쿼리를 실행하는 Redshift Spectrum을 포함
- 실시간 분석: Amazon Kinesis
    - IoT 원격 측정 데이터, 애플리케이션 로그 웹사이트 클릭스트림과 같은 스트리밍 데이터를 간편라게 수집, 처리, 분석할 수 있음
    - 모든 데이터가 수집된 후 처리하는 것이 아닌, 데이터 레이크에 데이터가 수신되는대로 처리, 분석하여 실시간으로 대응할 수 있음
- 운영 분석: Amazon Elasticsearch Service
    - 애플리케이션 모니터링, 로그 분석 및 클리스트림 분석 같은 운영 분석
    - 실시간으로 데이터를 검색, 탐색, 필터링 집계 및 시각화할 수 있음
    - Elasticsearch의 간편한 API 및 실시간 분석 기능과 더불어 프로덕션 워크로드에 필요한 가용성, 확장성, 보안성을 제공
- 대시보드 및 시각화: Amazon QuickSight
    - 빠르고 강력한 클라우드 기반 비즈니스 분석 서비스 제공
    - 모든 브라우저 또는 모바일 장치에서 액세스할 수 있는 멋진 시각화 및 풍부한 대시보드를 간편하게 작성 가능
